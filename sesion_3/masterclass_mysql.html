01 Bloque 1 · 30 min

Fundamentos y Modelo Relacional

¿Qué hay detrás de cualquier aplicación que guarda datos?

Bloque 1

¿Qué es una base de datos relacional?

Una base de datos relacional organiza la información en tablas con filas y columnas, relacionadas entre sí mediante claves.

Propuesta por Edgar F. Codd (IBM, 1970) y basada en la teoría de conjuntos.

El lenguaje estándar de consulta es SQL (Structured Query Language).

MySQL PostgreSQL SQLite Oracle SQL Server MariaDB

Conceptos clave

  • Tabla — entidad del mundo real
  • Fila / Registro — un dato concreto
  • Columna / Campo — atributo
  • Clave Primaria (PK) — identificador único
  • Clave Foránea (FK) — referencia a otra tabla
  • Esquema — estructura de la BD
Bloque 1

Propiedades ACID

Las transacciones en una BD relacional garantizan integridad mediante cuatro propiedades:

A — Atomicidad

Una transacción es todo o nada. Si falla un paso, se revierten todos los cambios.

C — Consistencia

La BD pasa de un estado válido a otro estado válido. No se rompen las reglas de integridad.

I — Aislamiento

Las transacciones concurrentes no se interfieren entre sí. Cada una ve la BD en un estado coherente.

D — Durabilidad

Una transacción confirmada (COMMIT) persiste aunque haya un fallo del sistema.

Bloque 1

Modelo Entidad-Relación (ER)

El modelo ER es una herramienta de diseño visual que permite definir la estructura de la BD antes de escribir código.

Componentes

  • Entidad — objeto del mundo real (Cliente, Producto)
  • Atributo — propiedad de la entidad (nombre, precio)
  • Relación — vínculo entre entidades (compra, pertenece)

Cardinalidades

  • 1:1 — Persona → DNI
  • 1:N — Cliente → Pedidos
  • N:M — Pedido ↔ Productos

DDL · DML · Tipos de datos · Claves · Constraints

Bloque 2 · DDL

Crear la base de datos y tablas

DDL (Data Definition Language) define la estructura de la BD.

SQL-- 1. Crear la base de datos
CREATE DATABASE tienda
  CHARACTER SET utf8mb4
  COLLATE utf8mb4_unicode_ci;

-- 2. Seleccionarla
USE tienda;

-- 3. Crear tabla clientes
CREATE TABLE clientes (
  id_cliente  INT          AUTO_INCREMENT PRIMARY KEY,
  nombre      VARCHAR(100) NOT NULL,
  email       VARCHAR(150) UNIQUE NOT NULL,
  telefono    VARCHAR(15),
  created_at  DATETIME     DEFAULT NOW()
);

Comandos DDL

  • CREATE — crear objetos
  • ALTER — modificar estructura
  • DROP — eliminar objeto
  • TRUNCATE — vaciar tabla
SQL-- Añadir una columna
ALTER TABLE clientes
  ADD COLUMN ciudad VARCHAR(80);

-- Eliminar tabla (¡cuidado!)
DROP TABLE IF EXISTS clientes;

💡 Reglas de oro

  • Usa DECIMAL para dinero, nunca FLOAT
  • VARCHAR para texto variable, CHAR para códigos fijos (ISO, DNI)
  • DATETIME para log de eventos; DATE para fechas de nacimiento
Bloque 2 · Integridad

Claves y restricciones

SQLCREATE TABLE productos (
  id_producto INT            AUTO_INCREMENT,
  nombre      VARCHAR(200)  NOT NULL,
  precio      DECIMAL(10,2) NOT NULL,
  stock       INT           DEFAULT 0,
  id_categoria INT,
  PRIMARY KEY (id_producto),
  CONSTRAINT chk_precio CHECK (precio > 0),
  CONSTRAINT chk_stock  CHECK (stock >= 0),
  FOREIGN KEY (id_categoria)
    REFERENCES categorias(id_categoria)
    ON DELETE SET NULL
    ON UPDATE CASCADE
);

Constraints disponibles

  • PRIMARY KEY — unicidad + no nulo
  • UNIQUE — valor único por fila
  • NOT NULL — campo obligatorio
  • DEFAULT — valor por defecto
  • CHECK — condición lógica
  • FOREIGN KEY — integridad referencial

ON DELETE / ON UPDATE

  • CASCADE — propaga el cambio
  • SET NULL — pone NULL en FK
  • RESTRICT — bloquea la operación

SELECT — consultar registros

SQL-- Todas las columnas
SELECT * FROM clientes;

-- Columnas específicas con filtro
SELECT nombre, email
FROM   clientes
WHERE  ciudad = 'Madrid'
ORDER BY nombre ASC
LIMIT 10;

-- Con alias
SELECT nombre AS cliente,
       precio * 1.21 AS precio_iva
FROM   productos
WHERE  precio > 50;
Bloque 2 · DML

Modificar y eliminar datos

UPDATE — modificar registros

SQL-- Actualizar un campo
UPDATE productos
SET    precio = 79.99,
       stock  = stock - 1
WHERE  id_producto = 1;

-- Actualizar múltiples filas
UPDATE productos
SET    precio = precio * 0.9
WHERE  stock > 100;

⚠️ Siempre usa WHERE en UPDATE y DELETE o modificarás todas las filas.

DELETE — eliminar registros

SQL-- Eliminar un registro
DELETE FROM clientes
WHERE id_cliente = 5;

-- Eliminar con condición
DELETE FROM pedidos
WHERE fecha < '2023-01-01';

-- Vaciar tabla (más rápido)
TRUNCATE TABLE logs_temp;

Bloque 3 · JOINs

JOINs — Combinar tablas

Tipos de JOIN

  • INNER JOIN — solo filas con coincidencia en ambas tablas
  • LEFT JOIN — todas las filas de la tabla izquierda (FROM); NULL si no hay coincidencia en la derecha
  • RIGHT JOIN — todas las filas de la tabla derecha (JOIN); NULL si no hay coincidencia en la izquierda
  • CROSS JOIN — producto cartesiano (raro)
SQL-- Clientes con sus pedidos (INNER)
SELECT c.nombre, p.fecha, p.total
FROM   clientes  c
INNER JOIN pedidos p
  ON c.id_cliente = p.id_cliente
ORDER BY p.fecha DESC;
SQL-- Clientes SIN pedidos (LEFT JOIN)
SELECT c.nombre, c.email
FROM   clientes  c
LEFT JOIN pedidos p
  ON c.id_cliente = p.id_cliente
WHERE  p.id_pedido IS NULL;

-- 3 tablas: pedido + cliente + producto
SELECT c.nombre, pr.nombre AS producto,
       d.cantidad, d.precio_unit
FROM   pedidos         p
JOIN   clientes        c  ON p.id_cliente  = c.id_cliente
JOIN   pedido_detalle  d  ON p.id_pedido   = d.id_pedido
JOIN   productos       pr ON d.id_producto = pr.id_producto
WHERE  p.estado = 'entregado';
Ejemplo visual — LEFT JOIN:
clientes (FROM · izquierda)
LEFT JOIN
pedidos (JOIN · derecha)
→ Devuelve todos los clientes; los que no tienen pedidos aparecen con NULL en las columnas de pedidos.

RIGHT JOIN hace lo mismo pero garantizando todos los registros de la tabla derecha. En la práctica se usa poco; es más legible invertir el orden y usar LEFT JOIN.

Bloque 3 · Agregación

GROUP BY & Funciones de agregación

Funciones de agregación

COUNT(*) Número de filas
SUM(col) Suma
AVG(col) Media aritmética
MAX(col) Valor máximo
MIN(col) Valor mínimo 
SQL-- Total de ventas por cliente
SELECT   c.nombre,
         COUNT(p.id_pedido)  AS num_pedidos,
         SUM(p.total)        AS total_gastado,
         AVG(p.total)        AS ticket_medio
FROM     clientes c
JOIN     pedidos  p ON c.id_cliente = p.id_cliente
GROUP BY c.id_cliente, c.nombre
ORDER BY total_gastado DESC;

HAVING — filtrar grupos

WHERE filtra filas. HAVING filtra grupos ya calculados.

SQL-- Clientes con más de 3 pedidos
SELECT   c.nombre,
         COUNT(*) AS total_pedidos
FROM     clientes c
JOIN     pedidos  p
         ON c.id_cliente = p.id_cliente
GROUP BY c.id_cliente, c.nombre
HAVING   COUNT(*) > 3
ORDER BY total_pedidos DESC;

Orden de ejecución SQL:
FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT

Bloque 3 · Subconsultas

Subconsultas

Una subconsulta es un SELECT anidado dentro de otra sentencia.

SQL-- Productos más caros que la media
SELECT nombre, precio
FROM   productos
WHERE  precio > (
  SELECT AVG(precio) FROM productos
);

-- Clientes que han comprado alguna vez
SELECT nombre
FROM   clientes
WHERE  id_cliente IN (
  SELECT DISTINCT id_cliente
  FROM   pedidos
  WHERE  estado = 'entregado'
);
SQL-- Subconsulta en FROM (tabla derivada)
SELECT nombre, total_gastado
FROM (
  SELECT c.nombre,
         SUM(p.total) AS total_gastado
  FROM   clientes c
  JOIN   pedidos  p
         ON c.id_cliente = p.id_cliente
  GROUP BY c.id_cliente, c.nombre
) AS resumen
WHERE total_gastado > 500;

¿Subconsulta o JOIN?

  • JOIN suele ser más eficiente
  • Subconsulta = más legible en ciertos casos
  • Usa EXISTS en lugar de IN para grandes datasets

CRUD completo desde Python

Python# CREATE — insertar con parámetros
def crear_cliente(conn, nombre, email):
    sql = """INSERT INTO clientes
              (nombre, email)
              VALUES (%s, %s)"""
    with conn.cursor() as cur:
        cur.execute(sql, (nombre, email))
        conn.commit()
        return cur.lastrowid

# READ
def obtener_clientes(conn):
    with conn.cursor(dictionary=True) as cur:
        cur.execute("SELECT * FROM clientes")
        return cur.fetchall()
Python# UPDATE
def actualizar_email(conn, cliente_id, nuevo_email):
    sql = """UPDATE clientes
              SET email = %s
              WHERE id_cliente = %s"""
    with conn.cursor() as cur:
        cur.execute(sql, (nuevo_email, cliente_id))
        conn.commit()

# DELETE
def eliminar_cliente(conn, cliente_id):
    sql = "DELETE FROM clientes WHERE id_cliente = %s"
    with conn.cursor() as cur:
        cur.execute(sql, (cliente_id,))
        conn.commit()

Bloque 4 · ORM

Introducción a SQLAlchemy

SQLAlchemy es el ORM estándar de Python. Permite trabajar con la BD usando clases Python en lugar de SQL puro.

terminalpip install sqlalchemy pymysql
Pythonfrom sqlalchemy import (
    create_engine, Column,
    Integer, String, Numeric
)
from sqlalchemy.orm import (
    declarative_base, Session
)

Base = declarative_base()

class Producto(Base):
    __tablename__ = "productos"

    id_producto = Column(Integer, primary_key=True)
    nombre      = Column(String(200), nullable=False)
    precio      = Column(Numeric(10,2))
    stock       = Column(Integer, default=0)
Python# Conexión
engine = create_engine(
    "mysql+pymysql://root:pass@localhost/tienda",
    echo=False
)

# Consultar
with Session(engine) as session:
    productos = (
        session.query(Producto)
               .filter(Producto.precio > 50)
               .order_by(Producto.precio.desc())
               .all()
    )
    for p in productos:
        print(p.nombre, p.precio)

# Insertar
with Session(engine) as session:
    nuevo = Producto(
        nombre="Auriculares BT",
        precio=59.99,
        stock=30
    )
    session.add(nuevo)
    session.commit()
¿Qué hace declarative_base()?

Crea una clase base de la que heredan todos los modelos. SQLAlchemy la usa como registro central: cada subclase le dice al ORM "existe una tabla con esta estructura". La asignación Base = declarative_base() solo ocurre una vez en el proyecto.

¿Necesito crear la tabla en Workbench?

No. Esta es una de las ventajas del ORM: defines la tabla como una clase Python y la creas en la BD con una sola línea. No hace falta tocar Workbench ni escribir CREATE TABLE a mano.

# Crea en la BD todas las tablas definidas como clases
Base.metadata.create_all(engine)

¿Qué hace @contextmanager?

Hace que get_connection() se pueda usar con with:

with get_connection() as conn:
    # aquí usas conn con seguridad

Python ejecuta la función en 3 fases automáticas:

  1. Abre la conexión — el código antes del yield
  2. Ejecuta tu código — el cuerpo del bloque with
  3. Cierra siempre — el finally, aunque haya un error en el paso 2

Sin esto tendrías que cerrar conn.close() tú mismo, y si el código falla a mitad, la conexión quedaría abierta.

¿Qué es DB_CONFIG?

Un diccionario con los datos de conexión a la BD. Se define una vez arriba del script y se reutiliza en cualquier parte.

El ** delante lo desempaqueta: en vez de escribir cada parámetro suelto, Python los pasa todos a la vez:

# Equivale a escribir esto:
mysql.connector.connect(
    host="localhost", user="root",
    password="****", database="tienda"
)
Python# Estructura sugerida del script
import mysql.connector
from contextlib import contextmanager

DB_CONFIG = {"host":"localhost", "user":"root",
             "password":"****", "database":"tienda"}

@contextmanager
def get_connection():
    conn = mysql.connector.connect(**DB_CONFIG)
    try:
        yield conn
    finally:
        conn.close()
